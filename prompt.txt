    def run_cleaner(

Behavior, in this order (order matters — later steps depend on earlier
ones):
1. Remove duplicate rows (remove_duplicate_rows) — always runs first.
2. Drop columns above drop_threshold missingness (drop_high_missingness_columns)
   — remove corresponding entries from `profiles` too, so downstream stages
   never see stale ColumnProfiles for dropped columns.
3. Drop NUMERIC_ID columns (drop_numeric_id_columns) — same profile
   pruning as above.
4. For each remaining column, based on its (possibly now-stale — re-check
   missing_rate from the profile, do not re-infer semantic_type per
   CLAUDE.md's "no re-inference" rule):
   - CATEGORICAL_LOW: normalize_category_values, then mode_impute if
     missing_rate < some reasonable working threshold (use 0.3, consistent
     with the outlier-handling missingness gate), else constant_fill.
     Add missing indicator column if missing_rate > 0.
   - CATEGORICAL_HIGH: normalize_category_values, group_rare_categories
     using profile.rare_categories, then constant_fill for missing (never
     mode — per PLAN.md's explicit per-strategy mapping).
   - NUMERIC_FEATURE: median_impute if missing_rate < 0.3, add missing
     indicator if missing_rate > 0. THEN — critically — only call
     winsorize_outliers if missing_rate <= 0.30, per CLAUDE.md's "Do not
     apply outlier handling to columns with missingness > 30%" rule. This
     is the one enforcement point for that constraint; make it explicit
     and add a ReportEntry with severity="info" noting outlier handling
     was skipped when the gate blocks it.
   - DATETIME_STRING: coerce_string_dates_to_datetime.
   - Other types (BOOLEAN, DATETIME_NATIVE, CONSTANT): pass through
     unchanged (CONSTANT columns should already be rare after drop steps,
     but handle gracefully if present).
5. Emit a ReportEntry (stage="cleaner") for every remediation action taken
   — column, action, rationale, severity, before_stats/after_stats (e.g.
   before/after missing_rate). Column drops get severity="warning" (info
   loss); everything else "info" unless it's something like near-total
   winsorization touching >10% of values, which is "warning".
6. Return the cleaned df, the pruned ColumnProfile list (reflecting
   surviving columns only), the ReportEntry list, and a dict of
   "transformer specs" — the values used per column (e.g. median used,
   mode used, rare category mapping) so engineer.py / assembler.py can
   later build sklearn-compatible fitted-parameter transformers rather
   than recomputing statistics differently at inference time.
